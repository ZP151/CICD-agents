# ADR-0005: Authentication and secret handling

- Status: Accepted
- Date: 2026-05-18

## Context

We need to authenticate to Azure DevOps (REST), Azure OpenAI, and (in v3)
GitHub. We also need to make sure no secret is ever written to logs, the
SQLite memory store, task results, or version control.

## Decision

- Local Dev Agent:
  - Azure DevOps PAT stored in the OS keyring under service `cicd-agent`,
    user `azure-devops-pat`; read at call time, never persisted.
  - Azure OpenAI key may live in `.env` (per-user) or in the keyring under
    service `cicd-agent`, user `azure-openai-key` (keyring takes priority).
  - `dev-agent configure-pat` and `dev-agent configure-aoai` are the only
    supported ways to set these locally.
- Cloud Review Agent:
  - User-assigned managed identity bound to the Container App.
  - Service Principal client secret in Key Vault (`kv-cicd-agent`).
  - At cold start the agent calls Key Vault for the secret, exchanges it
    for an ADO bearer token, caches the token in-memory until expiry.
- All captured subprocess output is run through a redaction pass that
  strips `Authorization`, `apikey`, `pat`, `password` patterns before it
  is persisted or returned via the API.

## Consequences

- Positive: secrets never touch disk in the cloud, never touch source
  control or task results locally; rotation is a Key Vault / keyring
  operation only.
- Negative: re-keying ADO requires the operator to update Key Vault and
  restart Container Apps revisions; documented in the runbook.

## Alternatives considered

- Long-lived ADO PAT in cloud env vars: rejected; cannot be rotated
  without redeploying and is hard to audit.
- Mounted file secrets: rejected because Container Apps' secret-mount
  story is less idiomatic than env vars + Key Vault reference.
